MySQL

1. MySQL 基础

1. SQL 分类

SQL 可以分为两个部分：数据定义语言（DDL）和数据操作语言（DML）。

  • DDL 部分的作用是创建/删除数据库或创建/删除表，也可以定义索引（键），规定表之间的链接，以及施加表间的约束。重要的 DDL 语句有：
    • CREATE DATABASE - 创建新数据库
    • ALTER DATABASE - 修改数据库
    • CREATE TABLE - 创建新表
    • ALTER TABLE - 变更（改变）数据库表
    • DROP TABLE - 删除表
    • CREATE INDEX - 创建索引（搜索键）
    • DROP INDEX - 删除索引
  • DML 部分的作用是对表中的数据进行增删改查。重要的 DML 语句有：
    • SELECT - 从数据库表中获取数据
    • UPDATE - 更新数据库表中的数据
    • DELETE - 从数据库表中删除数据
    • INSERT INTO - 向数据库表中插入数据

2. SQL 数据类型

  • Text 数据类型
使用方式
CHAR(size) 保存固定长度的字符串（可包含字母、数字以及特殊字符）。在括号中指定字符串的长度。最多 255 个字符。
VARCHAR(size) 保存可变长度的字符串（可包含字母、数字以及特殊字符）。在括号中指定字符串的最大长度。最多 255 个字符。注释：如果值的长度大于 255，则被转换为 TEXT 类型。
TINYTEXT 存放最大长度为 255 个字符的字符串。
TEXT 存放最大长度为 65,535 个字符的字符串。
  • Number 数据类型
数据类型 使用方式
TINYINT(size) -128 到 127 常规，0 到 255 无符号。在括号中规定最大位数。
INT(size) -2147483648 到 2147483647 常规，0 到 4294967295 无符号。在括号中规定最大位数。
FLOAT(size,d) 带有浮动小数点的小数字，在括号中规定最大位数，在 d 参数中规定小数点右侧的最大位数。
DOUBLE(size,d) 带有浮动小数点的大数字，在括号中规定最大位数，在 d 参数中规定小数点右侧的最大位数。
DECIMAL(size,d) 作为字符串存储的 DOUBLE 类型，允许固定的小数点。
  • Date 数据类型
数据类型 使用方式
DATE() 日期。格式：YYYY-MM-DD，注释：支持的范围是从 ‘1000-01-01’ 到 ‘9999-12-31’
DATETIME() 日期和时间的组合。格式：YYYY-MM-DD HH:MM:SS，注释：支持的范围是从 ‘1000-01-01 00:00:00’ 到 ‘9999-12-31 23:59:59’
TIME() 时间。格式：HH:MM:SS 注释：支持的范围是从 ‘-838:59:59’ 到 ‘838:59:59’
YEAR() 2 位或 4 位格式的年。注释：4 位格式所允许的值：1901 到 2155。2 位格式所允许的值：70 到 69，表示从 1970 到 2069。

  1. 最基本的使用方法
--用法--
    select
        字段列表/列名称
    from
        表名
    where
        条件列表
    group by
        分组字段
    having
        分组之后的条件
    order by
        排序
    limit
        分页限定

--举例--
SELECT LastName,FirstName FROM Persons
  1. DISTINCT 语句

在表中可能查出有重复值的答案，使用 DISTINCT 语句用于返回唯一不同的值。

--用法--
--SELEC后多加一个DISTINCT
SELECT DISTINCT 列名称 FROM 表名称

--举例--
SELECT Company FROM Orders
  1. WHERE 语句

有条件的从表中选取数据，条件列表。

--用法--
SELECT 列名称 FROM 表名称 WHERE 列 运算符 值

--举例--
--用单引号来环绕文本值，数值不要用引号--
SELECT * FROM Persons WHERE City='Beijing'
SELECT * FROM Persons WHERE Year>1965

--BETWEEN区间左闭右开--
SELECT * FROM Persons WHERE Year BETWEEN 1965 AND 2020

--模糊查询：LIKE--
--占位符：_ 表示单个任意字符；% 表示多个任意字符--
-- 查询姓马的有哪些？
SELECT * FROM student WHERE NAME LIKE '马%';
-- 查询姓名第二个字是化的人
SELECT * FROM student WHERE NAME LIKE "_化%";
-- 查询姓名是3个字的人
SELECT * FROM student WHERE NAME LIKE '___';
  1. 聚合函数

将一列数据作为一个整体，进行纵向的计算，聚合函数后可以起别名，后续的使用更加方便。

  1. count()：计算个数
  2. max()：计算最大值
  3. min()：计算最小值
  4. sum()：计算和
  5. avg()：计算平均值
  • 聚合函数的计算排除 null 值，一般情况下选择非空的列，如主键。也可以使用 IFNULL() 函数。
--计算个数--
--使用主键--
SELECT COUNT(id) FROM STU;
--使用 IFNULL() 函数--
SELECT COUNT(IFNULL(Math,0)) FROM STU;
--计算最大值，并起别名，AS可以省略--
SELECT MAX(Math) AS 数学第一名 FROM STU;
  1. GROUP BY 语句

分组查询：分组之后查询的字段，应该为 分组字段（其他字段没有意义） 聚合函数1 聚合函数2 …

--按性别分组，计算每个性别的人数，和数学平均分。显示 sex count avg 三列--
SELECT sex, COUNT(id), AVG(Math) FROM STU GROUP BY sex

--按性别分组，计算数学平均分，低于 70 分不参与计算--
SELECT sex, COUNT(id), AVG(Math) FROM STU WHERE Math>70 GROUP BY sex
  1. HAVING 语句

分组之后的条件，聚合函数后可以起别名，后续的使用更加方便。使用在 GROUP BY 之后：

--按性别分组，计算数学平均分，低于 70 分不参与计算，并且人数大于 2 才查询，小于 2 不查询--
SELECT sex, COUNT(id), AVG(Math) FROM STU WHERE Math>70 GROUP BY sex HAVING COUNT(id) > 2
--用起别名的方式简化--
SELECT sex, COUNT(id) AS 人数, AVG(Math) FROM STU WHERE Math>70 GROUP BY sex HAVING 人数 > 2
  • WHERE 和 HAVING 的区别：
    1. WHERE 在分组之前进行限定，如果不满足条件，则不参与分组。HAVING 在分组之后进行限定，如果不满足结果，则不会被查询出来。
    2. WHERE 后不可以跟聚合函数，HAVING 可以进行聚合函数的判断。
  1. ORDER BY 语句

ORDER BY 语句用于对结果集进行排序，默认按升序排序，如果希望按降序排序，可以使用 DESC 关键字。

--举例--
SELECT Company, Number FROM Table1 ORDER BY Number
SELECT * FROM Table1 ORDER BY Math ASC, English DESC
  1. LIMIT 语句

  • 语法：LIMIT 开始的索引，每页查询的条数；

  • 公式：开始的索引 = （当前页码 - 1）* 每页显示的条数

  • LIMIT 是 MySQL 特有的分页方式，其他数据库如 Oracle 不能使用。

--每页显示3条--
SELECT * FROM student LIMIT 0,3; -- 第1页        
SELECT * FROM student LIMIT 3,3; -- 第2页        
SELECT * FROM student LIMIT 6,3; -- 第3页
  1. AND 和 OR 等运算符

AND 和 OR 可在 WHERE 子语句中把两个或多个条件结合起来。

--可用运算符如下--
>  <  >=  <=  =  <>(!=)
BETWEEN ... AND   LIKE(模糊查询)   IS NULL
AND(&&)  OR(||)  NOT(!)

--举例--
SELECT * FROM Persons WHERE FirstName='Thomas' AND LastName='Carter'
SELECT * FROM Persons WHERE firstname='Thomas' OR lastname='Carter'
SELECT * FROM Persons WHERE (FirstName='Thomas' OR FirstName='William')
AND LastName='Carer'

5. 其他 DML 语句

  1. INSERT INTO 语句

INSERT INTO 语句用于向表格中插入新的行。

--用法--
--所有列的值都要插入--
INSERT INTO 表名称 VALUES (值1, 值2,....)

Update 语句用于修改表中的数据。

6. 约束

概念：对表中的数据进行限定，保证数据的正确性、有效性和完整性。

  1. 非空约束：NOT NULL，某一列的值不能为 NULL。
--1.创建表时添加非空约束
CREATE TABLE STU(id INT,NAME VARCHAR(20) NOT NULL)
--2.创建完表后再添加非空约束
ALTER TABLE STU MODIFY NAME VARCHAR(20) NOT NULL
--3.删除NAME的非空约束
ALTER TABLE STU MODIFY NAME VARCHAR(20)
  1. 唯一约束：UNIQUE，某一列的值不能重复。可以有 NULL 值，但只能由一个。
--1.创建表时添加唯一约束
CREATE TABLE STU(id INT,NAME VARCHAR(20) UNIQUE)
--2.创建完表后再添加唯一约束
ALTER TABLE STU MODIFY NAME VARCHAR(20) UNIQUE
--3.删除NAME的唯一约束
ALTER TABLE STU DROP INDEX NAME
  1. 主键约束：主键是表中记录的唯一标识，非空且唯一。
--1.创建表时添加主键约束
CREATE TABLE STU(id INT PRIMARY KEY,NAME VARCHAR(20))
--2.创建完表后再添加主键约束
ALTER TABLE STU MODIFY id INT PRIMARY KEY
--3.删除主键约束
ALTER TABLE STU DROP PRIMARY KEY
--4.自动增长：数值类型的列，主键值自动增长
CREATE TABLE STU(id INT PRIMARY KEY auto_increment,NAME VARCHAR(20))
--5.创建后添加自增
ALTER TABLE STU id INT auto_increment
--6.删除自增
ALTER TABLE STU MODIFY id INT
  1. 外键约束：使表与表之间产生关系，从而保证数据的正确性。
--0.用法
create table 表名(
    ....
    外键列
    constraint 外键名称 foreign key (外键列名称) references 主表名称(主表列名称)

--1.创建表时添加外键约束，是单独的一行，并不是跟在某个列之后，前一列后要加逗号
CREATE TABLE STU(id INT PRIMARY KEY, class_id INT, 
    CONSTRAINT stu_class_id FOREING KEY(class_id) REFERENCES class(id))

--2.创建表后添加外键
ALTER TABLE STU ADD CONSTRAINT stu_class_id FOREIGN KEY(class_id) REFERENCES class(id)

--3.删除外键
ALTER TABLE STU DROP FOREIGN KEY stu_class_id

--4.设置级联 ON UPDATE CASCADE / DELETE UPDATE CASCADE
ALTER TABLE STU ADD CONSTRAINT stu_class_id FOREIGN KEY(class_id) REFERENCES class(id) ON UPDATE CASCADE

7. 多表关系

  1. 多对一关系：如学生和班级，一个班级对应多个学生。实现方式：在多的一方建立外键，指向一的一方的主键。
  2. 多对多关系：如学生和课程。实现方式：借助第三张中间表，中间表至少包含两个字段，作为外键分别指向其他两张表的主键。
  3. 一对一关系：如学生和学号。在任意一方添加唯一外键，指向另一方主键。

8. 多表查询

多表查询时会产生笛卡尔积，即很多冗余且不正确的信息，因此需要一些方法来规避。

  1. 内连接：INNER JOIN / JOIN ON (条件)，仅查询符合条件的部分。
--用法 INNER 可以省略
SELECT STU.name, STU.gender,class.name FROM STU (INNER) JOIN class ON STU.'class_id' = class.'id'
  1. 左外连接：LEFT JOIN ON (条件)。查询左表所有的行，即使不符合条件。
--用法
SELECT STU.name, STU.gender,class.name FROM STU LEFT JOIN class ON STU.'class_id' = class.'id'
  1. 右外连接：RIGHT JOIN ON (条件)。查询右表所有的行，即使不符合条件。
--用法
SELECT STU.name, STU.gender,class.name FROM STU RIGHT JOIN class ON STU.'class_id' = class.'id'
  1. 子查询：嵌套查询
--子查询的结果可以是单行单列、多行单列和多行多列的
--单行单列演示
SELECT * FROM emp WHERE emp.`salary` = (SELECT MAX(salary) FROM emp); --这个子查询结果 最高工资 是单行单列的

--多行单列演示 使用 IN 函数
SELECT * FROM emp WHERE dept_id IN (SELECT id FROM dept WHERE NAME = '财务部' OR NAME = '市场部');

--多行多列演示 子查询结果作为一张虚拟表
SELECT * 
FROM 
    dept t1 ,
    (SELECT * FROM emp WHERE emp.`join_date` > '2011-11-11') t2 --子查询结果作为t2表
WHERE 
    t1.id = t2.dept_id;

2. 事务

1. 什么是事务

概念：事务是逻辑上的一组操作，要么同时成功，要么同时失败。

举例：事务最经典也经常被拿出来说例子就是转账了。假如小明要给小红转账1000元，这个转账会涉及到两个关键操作就是：将小明的余额减少1000元，将小红的余额增加1000元。万一在这两个操作之间突然出现错误比如银行系统崩溃，导致小明余额减少而小红的余额没有增加，这样就不对了。事务就是保证这两个关键操作要么都成功，要么都要失败。

使用步骤：

--开启一个事务，两种方式
BEGIN
START TRANSACTION

--进行一组操作
UPDATE Persons SET Money=Money-1000 WHERE Name='小明'
UPDATE Persons SET Money=Money+1000 WHERE Name='小红'

--如果成功，则提交
COMMIT

--如果出错，则回滚
ROLLBACK

2. 事务的四大特性

  1. 原子性：事务是最小的执行单位，不允许分割。事务的原子性确保动作要么全部成功，要么全部失败；
  2. 持久性：一个事务被提交或回滚之后，数据库会持久化的保存数据，即使数据库发生故障也不应该对其有任何影响。
  3. 隔离性：多个并发事务之间相互独立，并发访问数据库时，各并发事务之间的数据库是独立的。
  4. 一致性：执行事务的前后，数据总量保持一致（完整性保持稳定）。如小明和小红每人有 1000 块钱，小明给小红转了 1000 块钱，不管成功还是失败，两个人的总量仍然是 2000。

3. 并发事务导致的问题

在典型的应用程序中，多个事务并发运行，经常会操作相同的数据来完成各自的任务（多个用户对同一数据进行操作）。并发虽然是必须的，但可能会导致以下的问题：

  1. 脏读：并发事务在读取数据时，读取到了另一个事务还没有提交的数据。
  2. 不可重复读（虚读）：在同一个事务中，两次读取到的数据不一样。也就是说在这个事务两次读取之间，有另一个并发事务修改了这个数据，导致其两次读取到的数据不一样。
  3. 幻读：它发生在一个事务（T1）读取了几行数据，接着另一个并发事务（T2）插入了一些数据时。在随后的查询中，第一个事务（T1）就会发现多了一些原本不存在的记录，就好像发生了幻觉一样，所以称为幻读。在 MySQL 中不会出现幻读。
  4. 丢失修改：一个事务的修改被另一个并发事务修改覆盖，称为丢失修改。是不可重复读的特殊情况。

注意：

  1. 不可重复读和脏读的区别：脏读读取到的是另一个事务未提交的数据，不可重复读读取到的是前一个事务已提交的数据。
  2. 不可重复读和幻读的区别：两者读取到的都是前一个事务已提交的数据，但不可重复读读取到的是对某些数据的修改，幻读读取到的是记录的增加或减少。

4. 事务的隔离级别

  1. READ-UNCOMMITTED(读取未提交)： 最低的隔离级别，允许读取尚未提交的数据变更，可能会导致脏读、幻读或不可重复读。
  2. READ-COMMITTED(读取已提交)： 允许读取并发事务已经提交的数据，可以阻止脏读，但是幻读或不可重复读仍有可能发生。
  3. REPEATABLE-READ(可重复读)： 对同一字段的多次读取结果都是一致的，除非数据是被本身事务自己所修改，可以阻止脏读和不可重复读，但幻读仍有可能发生。
  4. SERIALIZABLE(可串行化)： 最高的隔离级别，完全服从ACID的隔离级别。所有的事务依次逐个执行，这样事务之间就完全不可能产生干扰，也就是说，该级别可以防止脏读、不可重复读以及幻读。
隔离级别 脏读 不可重复读 幻读
READ-UNCOMMITTED √ √ √
READ-COMMITTED × √ √
REPEATABLE-READ × × √
SERIALIZABLE × × ×
  • 随着隔离级别的递增，安全性越来越高，但效率越来越低。
  • MySQL 的默认隔离级别是 REPEATABLE-READ，Oracle 的默认隔离级别是 READ-COMMITTED。

3. 索引

1. 索引简介

  • 在不读取整个表的情况下，索引使数据库更快的查找数据，但使得读取相对变慢。
  • 索引可以分为聚集索引和非聚集索引：
    • 聚集索引存储物理上连续存在的记录，如123456；
    • 非聚集索引存储逻辑上的的连续记录，如某一作家的书可以在第一个和第十个书架上；
  • 主键和外键列一定要建立索引。
  • 经常查询的列最好建立索引，相对的不经常查询的列最好不要建立索引。
  • 经常存取的列避免建立索引。
--创建索引的语法
CREATE 
    INDEX index_name(索引名)
ON
    表名（列名）

--举例：在Person表中的姓名列建立索引
CREATE 
    INDEX personIndex
ON
    Persons(name)

2. 索引的存储方式

  1. 数据库索引是存储在外部磁盘上的，当数据库索引量很大时，就不能把整个索引加载到内存，只能逐一加载磁盘页，每一次加载都是一次 I/O 操作，这个过程耗时长；
  2. 二叉树当数据量很大时，深度会非常深。对于树来说，磁盘页数就是树的高度。若使用二叉树存储索引，读取时需要读取很多磁盘页，也就是要进行相当多的 I/O 操作，因此使用二叉树耗时长；
  3. 数据库索引的存储方式是 B/B+ 树，这是一种多叉树，它的每个节点最多包含 m 个孩子，m 被称为 B 树的阶，m 的大小取决于磁盘页的大小。这种树的特征是矮胖，可以减少 I/O 次数。

3. B树的结构

  1. B 树的每个节点最多包含 m 个孩子，m 被称为 B 树的阶，m 的大小取决于磁盘页的大小。
  2. B 树的存储方式符合搜索树的规范，节点左子树的数据都比当前节点小，右子树的数据都比当前节点大，且每个节点内部按非降序排列。
  3. 若根节点不是叶子节点，则根节点至少有两个孩子。
  4. 非叶子节点至少有 m/2 个孩子，最多有 m 个孩子。
  5. 非叶子节点有 k 个孩子，则非叶子节点中的数据个数等于 k - 1。如当前节点有 3 个数据，则该节点有 4 个孩子。k - 1 个数据分割 k 个孩子节点。
  6. 所有叶子节点处于同一层。

4. B树和B+树的对比

  1. B树每个节点中存储的数据都是键值对，即索引-数据对；B+树中非叶子节点只存储关键字，用来索引，叶子节点中存储了所有数据。
  2. B+树非叶子节点中数据个数为 k 个，B树为 k-1 个。
  3. B+树叶子节点有指向相邻节点的指针，按照关键字从小到大的顺序连接，形成一个有序链表。B树不是这种结构。
  4. B+树中同一个数字会在不同的节点中出现，父节点中会记录每个孩子中的最大或最小元素。

5. B+树的优势

  1. 单一节点中存储的数据更多，使得 B+树更加矮胖，查询的 I/O 次数更少；
  2. 所有的查询都要查询到叶子节点，查询性能稳定。B树每个节点都可能查到数据，所以不稳定。
  3. 叶子节点形成有序链表，便于范围查找。B树只能通过中序遍历查找，十分繁琐。

6. 哈希索引

哈希索引就是采用一定的哈希算法，把键值换算成新的哈希值，检索时不需要类似B+树那样从根节点到叶子节点逐级查找，只需一次哈希算法即可立刻定位到相应的位置，速度非常快。

哈希索引有好几个局限：

  • 哈希索引也没办法利用索引完成排序。
  • 不支持最左匹配原则。
  • 在有大量重复键值情况下，哈希索引的效率也是极低的—->哈希碰撞问题。
  • 不支持范围查询。

7. 索引最左匹配原则

  • 索引可以简单如一个列(a)，也可以复杂如多个列(a, b, c, d)，即联合索引。
  • 如果是联合索引，那么 key 也由多个列组成，同时，索引只能用于查找 key 是否存在（相等），遇到范围查询 (>、<、between、like 左匹配)等就不能进一步匹配了，后续退化为线性查找。
  • 因此，列的排列顺序决定了可命中索引的列数。
  • 例如：有索引 (a, b, c, d)，查询条件 a = 1 and b = 2 and c > 3 and d = 4，则会在每个节点依次命中 a、b、c，无法命中 d。(很简单：索引命中只能是相等的情况，不能是范围匹配) 。
  • 不需要考虑=、in等的顺序，mysql会自动优化这些条件的顺序，以匹配尽可能多的索引列。 例如有索引 (a, b, c, d)，查询条件 c > 3 and b = 2 and a = 1 and d < 4 与 a = 1 and c > 3 and b = 2 and d < 4 等顺序都是可以的，MySQL会自动优化为 a = 1 and b = 2 and c > 3 and d < 4，依次命中 a、b、c。

8. 聚集索引和非聚集索引

  • 聚集索引是索引结构和数据一起存放的索引，以主键创建的索引；
  • 非聚集索引是索引结构和数据分开存放的索引，以非主键创建的索引。

区别：

  • 聚集索引在叶子节点存储的是表中的数据；
  • 非聚集索引在叶子节点中存储的是主键和索引列信息；
  • 使用非聚集索引查询时，要先拿到主键，再通过主键去查询数据，这个过程也叫做回表。

9. 覆盖索引

我们知道在 InnoDB 存储引擎中，如果不是主键索引，叶子节点存储的是主键和索引列息。最终还是要回表，也就是要通过主键再查找一次。

  • 覆盖索引即需要查询的字段正好是索引的字段，那么直接根据该索引，就可以查到数据了， 而无需回表查询。

例如：

--创建单列索引username，进行查询时，只查询username不需要进行回表
select username from user where username = 'Java3y'
--如果既要查询username又要查询age，则需要进行回表
select username , age from user where username = 'Java3y'
--若想要不回表，可以将单列索引升级为联合索引(username,age)，再进行查询则不需要回表
select username , age from user where username = 'Java3y'

10. 不触发索引

  1. 条件中多个字段（列）使用 or，即使有条件带索引也不会使用。多条件查询统一字段是使用 or 索引会生效。可以用 in 代替 or。
  2. 多列索引 / 联合索引没有使用第一个或左边的索引，之后就不会使用索引。（最左匹配原则）
  3. 模糊查询 like 以 % 开头。
  4. 如果列类型是字符串，数据没有用引号（发生了类型转换）。
  5. 在索引列上使用 IS NULL / IS NOT NULL 操作。
  6. 索引上有数学运算。
  7. 索引上有函数运算。
  8. 查询条件里有不等号（NOT < !=）。
  9. MySQL 估计全表扫描更快时。

4. SQL 优化

1. 数据库字段设计规范

  1. 优先选择符合存储需要的最小数据类型；
  2. 尽可能把所有列都定义为 NOT NULL，因为 NULL 值需要额外空间保存且比较和计算时要进行特别的处理；
  3. 用 TIMESTAMP 或 DATETIME 存储时间，不要使用字符串；
  4. 同财务相关的数据必须使用 DECIMAL 类型，计算时不会丢失精度。

2. 索引设计规范

  1. 每张表的索引不要过多；
  2. 每个表都定义一个主键；
  3. 避免冗余索引和重复索引；
  4. 频繁查询的列考虑使用覆盖索引；
  5. 尽量避免使用外键。

3. 开发规范

  1. 避免会导致索引失效的操作语句；
  2. 禁止使用 * ；
  3. 避免使用子查询。由于子查询的结果集无法使用索引，所以子查询为简单语句时，可以优化为 join 操作；
  4. 在明显不会有重复值时，使用 UNION ALL 而不是 UNION。因为 UNION 会对结果集进行去重操作，UNION ALL 不会；
  5. 查询不要拿全表，可以用 LIMIT 分页，每页的数量也不要太大。

4. 大表优化

  1. 优化字段：如使用 TINYINT、SMALLINT、MEDIUM_INT 作为整数类型而非INT，如果非负则加上 UNSIGNED，用整数代替字符串，用整形存 IP 等；
  2. 优化索引：
    1. 根据查询有针对性的创建，查询多分布广的列适合建立索引，而如性别这种分布稀疏的列（只有男女两种）不适合建立索引。
    2. 索引也要控制数量，不是越多越好，删除不必要的单利索引。
    3. 多列索引注意顺序，将区分度高的（分布广的）列或使用最频繁的列放在左侧。
    4. 查询频繁的列使用覆盖索引。
    5. 不用外键和 UNIQUE 等，用程序保证约束。
  3. 引擎：MyISAM 适合读操作频繁的情况，InnoDB 适合增删改频繁的情况。
  4. SQL 优化：通过日志查找较慢的 SQL，避免会导致索引失效的操作语句，使用 LIMIT 分页等。
  5. 缓存：使用 Redis 等；
  6. 读写分离：主从复制，主库写从库读，尽量不要采用双主；
  7. 表分区：物理上将表分为多个区域，但逻辑上还是一张表。
  8. 垂直拆分：按列分为多张表，但会造成主键冗余，引起 JOIN 操作，事务处理复杂等问题。
  9. 水平拆分：按行拆分成多张表。若在库内分表，只是解决了单一表数据量过大的问题，对于减轻数据库压力来说没有什么作用。这就要通过分库解决。
