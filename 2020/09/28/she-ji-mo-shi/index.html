设计模式

1. 单例模式

1. 单例模式简介

单例模式就是保证一个类只有一个实例。

目的：频繁的创建一个类的对象、管理对象和销毁对象是十分耗费资源的，只创建一个对象来避免这些消耗。某些特殊情况下，要求一个类的对象只能有一个。

特点：关键在于构造器私有化。

  1. 构造函数私有化，用 private 修饰构造函数，使外界无法显示地创建对象。
  2. 在类的内部创建一个实例对象。
  3. 向外界提供一个得到实例对象的方法。

破坏：在特殊情况下，单例模式会被攻击破坏。

  1. 序列化与反序列化：利用输出流（OutputStream）与输入流（InputStream），先写出在读入，这时会产生一个新的对象。原因是在读入（InputStream），也就是反序列化时，会通过反射调用类的无参构造，创建出新的对象。
  2. 反射：反射可以通过 setAccessable(true) 获取私有构造器，通过构造器创建出多个不同的对象。上述序列化和反序列化其实也是用到反射。

解决方式：

  1. 序列化与反序列化：在单例模式的类中，添加一个 readResolve() 方法，返回单例对象。原因是反序列化过程中，会判读是否有 readResolve() 方法，如果有，使用该方法的返回值作为反序列化的返回值。

  1. 反射：使用枚举单例模式，因为枚举类型在底层屏蔽了反射调用。

2. 饿汉式单例模式

  • 在类的内部先创建出一个实例对象，向外界提供方法获取。
package singleton;

/**
 * @author 巨瑞康
 */
public class Hungry {

    //在类的内部创建私有对象，必须是静态变量
    private static Hungry single = new Hungry();

    //构造器私有化
    private Hungry(){

    }

    //对外提供的获得对象的方法，必须是静态方法，通过类名调用
    public static Hungry getInstance(){
        return single;
    }

    //内部的其他方法可以不是静态方法，通过获得的单例对象调用
    public void print(){
        System.out.println("饿汉式单例模式");
    }
}

缺点

  1. 类中先创建出了一个对象，在外部不需要使用这个实例对象时，是一种内存浪费。
  2. 不能保证线程安全。

2. 懒汉式单例模式

  • 不直接在类中创建实例对象，需要时先判断对象是否已创建，如果没有创建就先创建，如果已经创建就直接返回。
package singleton;

/**
 * @author 巨瑞康
 */
public class Lazy {

    //单例对象作为一个静态私有变量，不直接初始化
    private static Lazy lazy;

    //构造器私有化
    private Lazy() {

    }

    //对外提供的获得对象的方法，必须是静态方法，通过类名调用    
    public static Lazy getInstance() {
        if (lazy == null) {
            lazy = new Lazy1();
        }
        return lazy;
    }

    public void print(){
        System.out.println("懒汉式单例模式");
    }
}
  • 缺点：不能保证线程安全。

3. 双重校验锁单例模式

  • 在懒汉式单例模式的基础上加双重校验锁。
package singleton;

/**
 * @author 巨瑞康
 */
public class DCL {
    //对实例对象加volatile关键字保证可见性和有序性
    private static volatile DCL dcl;

    //构造器私有化
    private DCL(){

    }

    //采用静态代码块的方式，锁住类的Class对象，保证锁的唯一性
    public static DCL getInstance(){
        //第一个if判断是提高程序效率，已创建了对象直接返回，不用进同步代码块
        if(dcl == null){
            synchronized (DCL.class){
                //只有外层判断是不够的，可能会创建出多个对象，所以这里还要进行一次判断
                if(dcl == null){
                    dcl = new DCL();
                }
            }
        }
        return dcl;
    }

    public void print(){
        System.out.println("双重校验锁单例模式");
    }
}

特点

  • 一定要用 volatile 修饰实例对象变量，保证可见性和有序性（防止指令重排）。
  • 加锁前有一次非空判断，加锁后也有一次非空判断。
  • 保证线程安全（反射调用除外）。

4. 静态内部类单例模式

  • 静态内部类的特点，被调用时才进行初始化。
  • 保证线程安全。
  • 推荐使用。
package singleton;

/**
 * @author 巨瑞康
 */
public class InnerClass {
    //构造器私有化
    private InnerClass(){

    }

    //静态内部类，必须是静态才能被getInstance()访问到
    private static class Inner{
        //必须是静态才能通过类名访问到，而且用final修饰保证不变。
        private static final InnerClass IC = new InnerClass();
    }

    public static InnerClass getInstance(){
        return Inner.IC;
    }

    public void print(){
        System.out.println("静态内部类单例模式");
    }
}

5. 枚举单例模式

  • 反射可以通过 setAccessable(true) 获取私有构造器，通过构造器创建出多个不同的对象。
  • 以上的所有单例模式，都可以被反射破解，只有枚举单例模式不会被反射破解。
  • 枚举是 JDK 1.5 之后引入的新特性，其底层源码屏蔽了反射调用。
package singleton;

/**
 * @author 巨瑞康
 */
public enum Enum {
    //实例对象
    INSTANCE;

    public void print(){
        System.out.println("枚举式单例模式");
    }
}

2. 代理模式

1. 代理模式简介

  • 委托一个类作为另一个类的代理，将被代理类隐藏起来，为其他对象提供一种代理以控制对这个对象的访问。同时代理类可以有被代理类没有的方法，作为补充。
  • 举例：房产中介。房东只要卖房，交给中介代理，中介可以带房客看房，给房子打广告等等，这些都是不需要房东去做的。但最终的卖房一定是房东来执行。
  • 举例：经纪人。明星要接活动，交给经纪人代理，经纪人可以谈合作，谈价钱，打广告等等，这些都是不需要明星去做的。但最终的活动一定是明星来执行。
  • 不管代理类中有多少方法，被代理类中要执行的方法一定是被代理类自己执行。
  • 从例子中看，代理人就是帮被代理人处理其不想做的事，被代理人只做自己想做的。且代理人处理的是通用事务，对于多个被代理人都可以适用。
  • 放到 Java 中看，被代理类是核心业务，代理类是可以被剥离出的辅助方法的集合，代理类可以代理不同的被代理类，为其提供方法的补充。典型案例就是 spring 的 AOP，基于动态代理实现。

2. 静态代理

由程序员创建或由特定工具自动生成源代码，再对其编译。在程序运行前，代理类的 .class 文件就已经存在。

3. 动态代理

动态代理的代理类是动态生成的，不是由程序员写好的。在程序运行时，代理类的 .class 文件才被创建。

Spring 的 AOP 就是基于动态代理实现的。根据是否有接口，动态代理可以分为两类：

  • 基于接口的动态代理：由 JDK 提供的 Proxy 类实现。
  • 基于类的动态代理：由 cglib 实现。

4. 基于 Proxy 的动态代理

基于 Proxy 的动态代理，其被代理类必须实现某个接口。

主要使用的类和接口：

  • InvocationHandler：函数式接口，其中只有一个方法 invoke()。invoke() 方法返回值为一个 Object 类，共有三个参数：
    1. Object proxy：调用该方法的代理实例，即动态代理实例。
    2. method：动态代理实例调用的方法。
    3. Object[] args：method 方法的参数。
  • Proxy：代理类，主要方法为 newProxyInstance() ，该方法返回一个 Object 动态代理实例，需要用接口（不能是实现类）接收，该方法有三个参数：
    1. ClassLoader loader：可以传入被代理类的 ClassLoader ，也可以传入实现 InvocationHandler 接口的类的 ClassLoader。一般只要没有自己定义 ClassLoader，传入的都是 AppClassLoader。
    2. interface：被代理类实现的接口。
    3. InvocationHandler h：实现 InvocationHandler 类的实例对象。

分步整理：

  1. 实现 InvocationHandler 接口。可以定义一个类，也可以用 lamda 表达式，重写其 invoke() 方法。最终动态代理实例调用的方法都会被拦截，转而执行 invoke() 方法。
  2. 调用 Proxy 类的静态方法 newProxyInstance()。将上一步实现 InvocationHandler 接口的实例，作为第三个参数传入。
  3. 接收 newProxyInstance() 方法的返回值，用接口接收，即为动态代理实例，可以通过这个实例调用（被代理类实现的）接口中的方法。

用类实现 InvocationHandler 接口：

package 代理模式;

import java.lang.reflect.InvocationHandler;
import java.lang.reflect.Method;
import java.lang.reflect.Proxy;

/**
 * @author 巨瑞康
 */
public class DynamicProxy implements InvocationHandler {

    //把被代理对象隐藏起来
    Agent agent = new Agent();

    //向外界提供获得动态代理实例的方法，就是调用了Proxy类的静态方法newProxyInstance()
    public Object getProxy(){
        //这里传入的参数分别是：
        //1. 当前类，也就是InvocationHandler接口实现类的ClassLoader
        //2. 被代理类实现的接口
        //3. 当前类，也就是InvocationHandler接口实现类
        return Proxy.newProxyInstance(this.getClass().getClassLoader(),agent.getClass().getInterfaces(),this);
    }

    //重写invoke方法，动态代理实例执行的所有方法都会被拦截，转而执行invoke方法，所以要实现的逻辑如添加前置方法和后置方法，都要在invoke中实现。
    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        System.out.println("拦截到" + method.getName() + "方法");
        before();
        Object obj = method.invoke(agent,args);
        after();
        return obj;
    }

    public void before(){
        System.out.println("代理类执行前置方法");
    }

    public void after(){
        System.out.println("代理类执行后置方法");
    }
}

主方法调用：

package 代理模式;

import java.lang.reflect.Proxy;

/**
 * @author 巨瑞康
 */
public class Test {
    public static void main(String[] args) {

        //首先获得实现InvocationHandler接口类的实例，我们在这个类中提供了获得动态代理实例的方法
        DynamicProxy dp = new DynamicProxy();

        //通过dp获得动态代理实例，用接口business接收
        business proxy = (business) dp.getProxy();

        //通过动态代理实例调用方法
        proxy.doBusiness1();
        System.out.println("================");
        proxy.doBusiness2();
    }
}

5. 基于 cglib 的动态代理

基于 cglib 的动态代理，其被代理类不需要实现接口

  1. 其原理是生成了一个子类，去继承被代理类，从而进行代理。
  2. 缺点：由于继承的局限性，被 final 修饰的类不能被代理，被代理类中的被 final 修饰的方法不能被重写。
  3. 导包：不使用 Spring 或 maven，只是测试使用的话，需要手动导入 cglib 的 jar 包。而 cglib.jar 会依赖一些其他的包如 asm 包，没有导入的话会报错，这两个包之间的版本不匹配也会报错，所以不建议使用。最好的方法是使用 cglib-nodep 包，不需要依赖，直接就可以使用，我使用的是 cglib-nodep-3.3.0.jar。

主要使用的类和接口：

  • MethodInterceptor：拦截器接口，类似 JDK 中的 InvocationHandler。实现这个接口，重写其中的 intercept() 方法。intercept() 方法返回值为一个 Object 类，共有四个参数：
    1. Object o：cglib 生成的代理类对象。
    2. Method method：被拦截的被代理类的方法。
    3. Object[] objects：方法的参数。
    4. MethodProxy methodProxy：代理方法。
  • Enhancer：提供生成代理类对象的方法。有三个必要的方法：
    • setSuperClass：设置 enhancer 对象的父类，参数类型是 .class。
    • setCallback：设置回调对象，参数类型是拦截器接口的实现类对象。
    • create：生成代理类实例对象，需要用被代理类接收。

实现 MethodInterceptor 接口

package 代理模式;

import net.sf.cglib.proxy.MethodInterceptor;
import net.sf.cglib.proxy.MethodProxy;
import java.lang.reflect.Method;

/**
 * @author 巨瑞康
 */
public class MyInterceptor implements MethodInterceptor{

    @Override
    public Object intercept(Object o, Method method, Object[] objects, MethodProxy methodProxy) throws Throwable {
        before();

        //这里是通过代理方法去调用父类的方法执行
        Object object = methodProxy.invokeSuper(o,objects);

        after();
        return object;
    }

    public void before(){
        System.out.println("cglib代理类执行前置方法");
    }

    public void after(){
        System.out.println("cglib代理类执行后置方法");
    }
}

主方法内生成代理类

package 代理模式;

import net.sf.cglib.proxy.Enhancer;

/**
 * @author 巨瑞康
 */
public class Test {
    public static void main(String[] args) {

        //获得一个Enhancer对象
        Enhancer enhancer = new Enhancer();

        //给enhancer对象设置父类，这里不需要父类的对象，直接用类名.class即可
        enhancer.setSuperclass(Agent.class);

        //给enhancer对象设置回调对象，即自定义拦截器
        enhancer.setCallback(new MyInterceptor());

        //利用enhancer对象生成代理实例对象，用父类接收
        Agent cglib = (Agent) enhancer.create();

        //通过代理实例调用方法
        cglib.doBusiness1();
        System.out.println("================");
        cglib.doBusiness2();
    }

3. 工厂模式

1. 工厂模式简介

工厂模式的作用

  1. 工厂模式最主要的作用就是解耦，当一个类需要另一个类的实例对象时，不需要显式的 new 一个对象，而是通过工厂进行生产，直接从工厂中获得，这样就减少了两个类之间的耦合性。
  2. 当需求改变时，不用修改代码，只要从工厂中取需要的实例对象即可。

工厂模式的分类

  1. 简单/静态工厂模式
  2. 工厂方法模式
  3. 抽象工厂模式

2. 简单工厂模式

简介

已知一些类，创建一个工厂，通过接收参数判断是这些类中的哪一个，然后创建出对应的实例对象。

代码实现

  1. Product 目标为手机，因此至少需要准备三件事：手机接口，实现类小米手机和苹果手机。
/**
 * @author 巨瑞康
 */
public interface Phone {
    /**
     * 手机接口
     */
    void thisPhone();
}
/**
 * @author 巨瑞康
 */
public class IPhone implements Phone{
    @Override
    public void thisPhone() {
        System.out.println("这是一台苹果手机");
    }

    public IPhone() {
        System.out.println("生产了一台苹果手机");
    }
}
/**
 * @author 巨瑞康
 */
public class MiPhone implements Phone {
    @Override
    public void thisPhone() {
        System.out.println("这是一台小米手机");
    }

    public MiPhone() {
        System.out.println("生产了一台小米手机");
    }
}
  1. 就当前的两个手机类创建一个工厂
/**
 * @author 巨瑞康
 */
public class PhoneFactory {
    public Phone makePhone(String name) {
        if ("MiPhone".equalsIgnoreCase(name)) {
            return new MiPhone();
        } else if ("iPhone".equalsIgnoreCase(name)) {
            return new MiPhone();
        } else {
            return null;
        }
    }
}
  1. 测试：
/**
 * @author 巨瑞康
 */
public class Test {
    public static void main(String[] args) {
        PhoneFactory pf = new PhoneFactory();
        Phone mi = pf.makePhone("miphone");
        Phone apple = pf.makePhone("iphone");
        mi.thisPhone();
        apple.thisPhone();
    }
}

3. 工厂方法模式

简介

工厂方法模式是指建立一个抽象工厂，将生产产品的任务分配给具体的工厂实现类，每一个产品有一个对应的工厂。

代码实现

  1. 创建一个工厂类接口，并对不同的产品创建不同的工厂类。
/**
 * @author 巨瑞康
 */
public interface AbstractFactory {
    /**
     * 抽象工厂
     */
    Phone makePhone();
}
/**
 * @author 巨瑞康
 */
public class MiFactory implements AbstractFactory {
    @Override
    public Phone makePhone() {
        return new MiPhone();
    }
}
/**
 * @author 巨瑞康
 */
public class AppleFactory implements AbstractFactory {
    @Override
    public Phone makePhone() {
        return new IPhone();
    }
}
  1. 测试：
/**
 * @author 巨瑞康
 */
public class Test {
    public static void main(String[] args) {
        AbstractFactory miFactory = new MiFactory();
        AbstractFactory appleFactory = new AppleFactory();
        Phone mi = miFactory.makePhone();
        Phone apple = appleFactory.makePhone();
        mi.thisPhone();
        apple.thisPhone();
    }
}

输出：

生产了一台小米手机
生产了一台苹果手机
这是一台小米手机
这是一台苹果手机

4. 抽象工厂模式

简介

抽象工厂模式通过在抽象工厂中增加创建产品的接口，并在子工厂中实现新产品的创建，达到一个工厂生产多种产品的效果。

代码实现

  1. 创建电脑接口，实现类小米电脑和苹果电脑。

 * @author 巨瑞康

     * 电脑接口

5. 三种模式对比

  1. 简单工厂模式：
    • 优点：一个工厂创建多种产品，代码量少，实现简单。
    • 缺点：需求改变时，如增加一个华为手机，首先增加一个华为手机实现类，但需要改变工厂类中的代码（如增加判断是否是华为手机）。这点可以用反射来改善。
  2. 工厂方法模式：
    • 优点：当需求改变时，如增加一个华为手机，只需要增加对应的类或工厂类，不需要改动已有的代码，易维护，增强系统的拓展性。
    • 缺点：每有一个产品就需要一个对应的工厂类，代码量大。
  3. 抽象方法模式：
    • 优点：多了一层抽象，将多个产品放在一个工厂中生产，减少了工厂的数量。
    • 缺点：复杂，难于扩展。如需要生产平板，就要修改抽象工厂，修改每个工厂实现类。
